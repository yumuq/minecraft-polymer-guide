# 01 - 核心概念

**文档版本：** 2.0.0  
**最后更新：** 2026-07-23

---

## 什么是 Polymer？

### 简单定义

**Polymer** 是一个 Fabric 服务器端库，允许服务器使用自定义模组内容（方块、物品、实体等），同时让原版客户端看到原版等效物。服务器实时将模组内容转换为原版表示。

### 核心机制

```
服务器端（安装模组）          客户端（原版）
┌─────────────────┐          ┌─────────────────┐
│ 自定义潜影盒方块  │  ────►  │  原版潜影盒方块   │
│ (mod:shulker)   │  Polymer │ (minecraft:     │
│                 │  转换层   │  shulker_box)   │
└─────────────────┘          └─────────────────┘
     服务器逻辑                   客户端渲染
```

**关键特性：**
- ✅ 服务器端保持完整的模组功能
- ✅ 客户端看到原版兼容的内容
- ✅ 无需客户端安装任何模组
- ⚠️ 客户端视觉效果受限（显示原版材质）

## 为什么需要 Polymer？

### 问题背景

**传统模组服务器的限制：**

```
场景 1：服务器安装了自定义方块模组
┌──────────────────────────────────────────┐
│ 服务器端注册表：                            │
│ - minecraft:stone                        │
│ - minecraft:dirt                         │
│ - custommod:magic_block  ← 自定义方块      │
└──────────────────────────────────────────┘
                  ↓
        客户端尝试连接
                  ↓
┌──────────────────────────────────────────┐
│ 客户端注册表：                             │
│ - minecraft:stone                        │
│ - minecraft:dirt                         │
│ ❌ custommod:magic_block 不存在！          │
└──────────────────────────────────────────┘
                  ↓
        RemapException
                  ↓
         连接失败！❌
```

**结果：**
- ❌ 原版客户端无法连接
- ❌ 必须强制所有玩家安装模组
- ❌ 服务器更新模组 = 所有玩家必须同步更新

---

### Polymer 的解决方案

```
场景 2：服务器安装了 Polymer 兼容的自定义方块模组
┌──────────────────────────────────────────┐
│ 服务器端注册表：                            │
│ - minecraft:stone                        │
│ - minecraft:dirt                         │
│ - custommod:magic_block                  │
│   └─ Polymer: 伪装为 minecraft:stone      │
└──────────────────────────────────────────┘
                  ↓
        客户端尝试连接
                  ↓
        Polymer 拦截同步
                  ↓
┌──────────────────────────────────────────┐
│ 客户端接收的注册表：                        │
│ - minecraft:stone                        │
│ - minecraft:dirt                         │
│ ✅ 只发送原版注册表项                       │
└──────────────────────────────────────────┘
                  ↓
         连接成功！✅
                  ↓
┌──────────────────────────────────────────┐
│ 游戏中：                                   │
│ 服务器：custommod:magic_block             │
│ 客户端看到：minecraft:stone（伪装）         │
│ 功能：服务器端完整逻辑正常运行              │
└──────────────────────────────────────────┘
```

**结果：**
- ✅ 原版客户端可以连接
- ✅ 无需强制安装模组
- ✅ 服务器功能完整保留
- ⚠️ 客户端视觉效果是原版材质

## Polymer 的工作原理

### 三层架构

```
┌─────────────────────────────────────────────────────────┐
│                     第 1 层：注册表隔离                      │
├─────────────────────────────────────────────────────────┤
│ 服务器启动时：                                             │
│ 1. 模组注册自定义内容到服务器注册表                         │
│ 2. Polymer 标记哪些内容需要隐藏                            │
│ 3. 构建两份注册表：                                        │
│    - 完整注册表（服务器使用）                               │
│    - 原版注册表（客户端同步）                               │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                     第 2 层：内容伪装                        │
├─────────────────────────────────────────────────────────┤
│ 运行时：                                                   │
│ 1. 服务器使用真实的模组内容                                │
│ 2. 发送给客户端前，Polymer 转换：                          │
│    custommod:magic_block → minecraft:stone              │
│    custommod:magic_sword → minecraft:diamond_sword      │
│ 3. 客户端接收原版内容，正常渲染                             │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                   第 3 层：数据过滤                          │
├─────────────────────────────────────────────────────────┤
│ NBT 数据同步时：                                           │
│ 1. 服务器保存完整的 NBT 数据（包括自定义数据）              │
│ 2. 同步给客户端前，Polymer 过滤：                          │
│    - 移除客户端不认识的自定义数据                           │
│    - 保留原版兼容的数据                                    │
│ 3. 客户端接收干净的原版 NBT                                │
└─────────────────────────────────────────────────────────┘
```

## 关键概念

### 1. 注册表同步（Registry Sync）

**什么是注册表？**
Minecraft 使用注册表管理所有游戏内容：
- 方块注册表：所有方块类型
- 物品注册表：所有物品类型
- 实体注册表：所有实体类型
- 菜单注册表：所有 GUI 类型

**同步过程：**
```
客户端连接时：
1. 服务器发送自己的注册表列表
2. 客户端对比自己的注册表
3. 如果不匹配 → RemapException → 断开连接
4. 如果匹配 → 继续连接
```

**Polymer 的干预：**
```java
// 没有 Polymer：
服务器注册表发送给客户端：
- minecraft:stone
- minecraft:dirt
- custommod:magic_block  ← 客户端没有 → ❌ 失败

// 有 Polymer：
PolymerBlockUtils.registerBlockEntity(CUSTOM_BLOCK_ENTITY);
PolymerMenuUtils.registerType(CUSTOM_MENU_TYPE);

服务器注册表发送给客户端：
- minecraft:stone
- minecraft:dirt
// custommod:magic_block 被隐藏了 → ✅ 成功
```

---

### 2. 方块/物品伪装

**实现方式：**
```java
public class CustomBlock extends Block implements PolymerBlock {
    @Override
    public BlockState getPolymerBlockState(BlockState state, PacketContext context) {
        // 客户端看到的方块
        return Blocks.STONE.defaultBlockState();
    }
}
```

**效果：**
- 服务器：存储和处理 `custommod:magic_block`
- 客户端：看到 `minecraft:stone`
- 功能：服务器端逻辑完全正常

**属性保留：**
```java
public BlockState getPolymerBlockState(BlockState state, PacketContext context) {
    BlockState vanillaState = Blocks.ANVIL.defaultBlockState();
    
    // 保留朝向属性，确保客户端渲染正确
    if (vanillaState.hasProperty(AnvilBlock.FACING)) {
        vanillaState = vanillaState.setValue(
            AnvilBlock.FACING,
            state.getValue(AnvilBlock.FACING)
        );
    }
    
    return vanillaState;
}
```

---

### 3. NBT 数据过滤

**问题：**
自定义方块实体可能存储客户端不理解的数据。

**解决：**
```java
// 注册需要 NBT 过滤的方块实体
PolymerBlockUtils.registerBlockEntity(CUSTOM_BLOCK_ENTITY_TYPE);

// Polymer 自动过滤同步给客户端的 NBT
// 服务器保留完整数据，客户端只接收原版兼容数据
```

---

### 4. 菜单类型隐藏

**问题：**
自定义 GUI 菜单类型会尝试同步到客户端。

**解决：**
```java
// 告诉 Polymer 不要同步这个菜单类型
PolymerMenuUtils.registerType(CUSTOM_MENU_TYPE);

// 客户端收到的菜单：
// - 可以是原版菜单类型
// - 可以是动态创建的虚拟菜单
// 服务器端逻辑不受影响
```

## 适配价值评估

### 何时值得适配？

✅ **推荐适配的情况：**

1. **服务器端功能模组**
   - 添加服务器端逻辑（如自定义合成）
   - 自定义方块/物品有服务器端行为
   - 原版客户端可以完整使用功能
   - **例子：** Easy Anvils（自定义铁砧方块）

2. **可选增强模组**
   - 模组客户端有增强体验
   - 原版客户端可以基本使用
   - 不影响游戏平衡
   - **例子：** Quick Shulker（右键功能仍可用）

3. **装饰性内容**
   - 自定义方块/物品用于装饰
   - 原版客户端看到原版等效物
   - 功能不受影响
   - **例子：** 自定义家具模组

---

⚠️ **需要谨慎评估的情况：**

1. **客户端驱动的功能模组**
   - 主要功能由客户端触发（快捷键、UI）
   - 原版客户端只能使用部分功能
   - 需要向用户明确说明限制
   - **例子：** Quick Shulker（快捷键不可用）

2. **视觉依赖的模组**
   - 自定义材质/模型是核心体验
   - 原版客户端看到原版材质
   - 可能影响用户体验
   - **例子：** 自定义盔甲显示模组

3. **复杂交互模组**
   - 需要客户端了解特殊机制
   - 原版客户端可能困惑
   - 需要详细的说明文档
   - **例子：** 自定义魔法系统

---

❌ **不建议适配的情况：**

1. **纯客户端模组**
   - 只在客户端运行（如小地图）
   - 没有服务器端逻辑
   - 适配没有意义
   - **例子：** JEI, Optifine

2. **核心游戏机制改变**
   - 彻底改变游戏玩法
   - 原版客户端无法理解
   - 会导致混乱
   - **例子：** 全新的战斗系统

3. **重度依赖客户端渲染**
   - 自定义渲染器
   - 特殊视觉效果
   - 原版客户端体验极差
   - **例子：** 自定义着色器模组

## 成本与收益分析

### 适配成本

| 模组类型 | 时间成本 | 技术难度 | 维护成本 |
|---------|---------|---------|---------|
| **仅菜单注册** | 10 分钟 | ⭐ 极简 | 极低 |
| **简单方块伪装** | 30 分钟 | ⭐⭐ 简单 | 低 |
| **复杂方块+实体** | 1-2 小时 | ⭐⭐⭐ 中等 | 中 |
| **多系统集成** | 半天+ | ⭐⭐⭐⭐ 复杂 | 高 |

---

### 适配收益

✅ **直接收益：**
- 原版玩家可以加入服务器
- 降低玩家进入门槛
- 扩大潜在玩家群体
- 减少客户端维护负担

✅ **间接收益：**
- 服务器更新灵活（不强制所有玩家同步）
- 可以A/B测试新功能
- 支持混合玩家群体
- 提升服务器兼容性声誉

## 常见误解

### ❌ 误解 1："Polymer 让原版客户端拥有模组功能"

**现实：**
- Polymer 只解决**连接问题**
- 功能可用性取决于**模组架构**
- 服务器端驱动 = 原版客户端可用
- 客户端驱动 = 原版客户端不可用

**例子：**
- Easy Anvils：右键方块 → 服务器处理 → ✅ 原版可用
- Quick Shulker：按快捷键 → 客户端发包 → ❌ 原版不可用

---

### ❌ 误解 2："所有模组都应该适配 Polymer"

**现实：**
- 纯客户端模组没必要（如 JEI）
- 重度依赖视觉的模组收益低
- 核心玩法改变的模组不适合

**判断标准：**
- 有服务器端逻辑？ → 考虑适配
- 只是客户端增强？ → 不适配
- 改变核心机制？ → 谨慎评估

---

### ❌ 误解 3："Polymer 适配很复杂"

**现实：**
- 简单模组（仅菜单）：10 分钟
- 中等模组（方块伪装）：1 小时
- 复杂模组（多系统）：半天

**关键：**
- 90% 的工作是**理解模组架构**
- 代码修改通常很少（< 50 行）
- 有模板可以直接使用

---

### ❌ 误解 4："原版客户端体验一定很差"

**现实：**
- 取决于模组类型
- 服务器端模组：体验 90%+
- 混合型模组：体验 50-80%
- 客户端驱动模组：体验 < 30%

**Easy Anvils 例子：**
- 原版客户端看到原版铁砧材质
- 但所有功能完全可用（物品存储、合成等）
- 体验 ≈ 95%

**Quick Shulker 例子：**
- 原版客户端可以右键打开
- 但无法使用快捷键
- 体验 ≈ 60%

## 总结

### 核心要点

1. **Polymer 是什么？**
   - 服务器端库
   - 隔离注册表
   - 伪装模组内容为原版

2. **解决什么问题？**
   - 注册表不匹配
   - 原版客户端无法连接
   - 强制模组安装

3. **如何工作？**
   - 注册表隔离
   - 内容伪装
   - NBT 过滤

4. **何时使用？**
   - 服务器端功能模组 ✅
   - 可选增强模组 ⚠️
   - 纯客户端模组 ❌
