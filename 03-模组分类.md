# 03 - 模组分类与适配策略

**文档版本：** 2.0.0  
**最后更新：** 2026-07-23

---

## 为什么需要分类？

**核心问题：** 不同类型的模组，Polymer 适配后原版客户端的功能可用性**完全不同**。

### 两个实战案例的对比

| 项目 | Easy Anvils | Quick Shulker |
|------|-------------|---------------|
| **适配难度** | ⭐⭐⭐ 中等 | ⭐ 极简 |
| **代码修改** | ~100 行 | ~10 行 |
| **原版客户端功能** | ✅ 100% 可用 | ⚠️ ~60% 可用 |

**为什么差异如此巨大？** → 因为它们是**不同类型**的模组！

---

## 模组分类体系

### 分类维度 1：功能触发方式

```
┌─────────────────────────────────────────────────────┐
│                   功能触发方式                         │
├─────────────────────────────────────────────────────┤
│                                                     │
│  服务器端触发              混合触发              客户端触发  │
│  ↓                       ↓                       ↓  │
│  右键方块                  右键 + 快捷键            快捷键    │
│  放置方块                  使用物品 + UI           UI操作   │
│  实体交互                  命令 + 按键             按键     │
│                                                     │
│  原版协议 ✅               部分原版协议 ⚠️           自定义协议 ❌ │
└─────────────────────────────────────────────────────┘
```

### 分类维度 2：数据存储位置

```
┌─────────────────────────────────────────────────────┐
│                   数据存储位置                         │
├─────────────────────────────────────────────────────┤
│                                                     │
│  服务器端数据              共享数据              客户端数据  │
│  ↓                       ↓                       ↓  │
│  方块实体NBT              物品NBT                客户端配置  │
│  世界存档                  玩家数据                本地缓存   │
│  服务器配置                同步数据                渲染状态   │
│                                                     │
│  Polymer完全支持 ✅        Polymer过滤 ⚠️          Polymer无关 ❓ │
└─────────────────────────────────────────────────────┘
```

## 四大模组类型

### 类型 1：纯服务器端模组

**定义：** 所有功能逻辑都在服务器端执行，客户端只负责渲染和接收结果。

**特征：**
- ✅ 使用原版游戏交互方式（右键、放置等）
- ✅ 数据存储在服务器端
- ✅ 客户端只需要显示结果
- ✅ 不需要自定义客户端代码

**典型例子：** Easy Anvils

**工作流程：**
```
原版客户端                     服务器（有模组）
     │                             │
     │ 1. 右键自定义铁砧方块           │
     ├──────────────────────────────►│
     │    (使用原版协议)               │ 2. 服务器接收
     │                                │ 3. 执行模组逻辑
     │                                │ 4. 处理物品存储
     │ 5. 接收容器GUI                 │
     │◄───────────────────────────────┤
     │    (原版菜单数据包)              │
     │                                │
     │ 6. 显示GUI                     │
     │ 7. 操作物品                    │
     ├──────────────────────────────►│
     │                                │ 8. 服务器更新状态
     │                                │
```

**Polymer 适配后：**
- ✅ 原版客户端：**100% 功能可用**
- ✅ 交互方式：完全原版
- ✅ 视觉效果：原版材质（但功能完整）

**适配难度：** ⭐⭐⭐ 中等（需要方块伪装）

---

### 类型 2：混合驱动模组

**定义：** 部分功能由服务器触发，部分功能由客户端触发。

**特征：**
- ⚠️ 基础功能使用原版交互
- ⚠️ 高级功能需要客户端代码
- ⚠️ 原版客户端可用基础功能
- ⚠️ 模组客户端可用全部功能

**典型例子：** Quick Shulker

**功能分层：**
```
┌─────────────────────────────────────────────┐
│          Quick Shulker 功能层次               │
├─────────────────────────────────────────────┤
│                                             │
│  第 1 层：基础右键功能                        │
│  - 右键手持潜影盒 → 打开                      │
│  - 右键手持末影箱 → 打开                      │
│  - 右键手持工作台 → 打开                      │
│  触发：原版右键事件                           │
│  处理：服务器端                              │
│  原版客户端：✅ 可用                          │
│                                             │
├─────────────────────────────────────────────┤
│                                             │
│  第 2 层：快捷键功能                          │
│  - 按键 → 扫描背包 → 快速打开                 │
│  - 按键 → 打开玩家背包                        │
│  触发：客户端快捷键                           │
│  处理：客户端发包 → 服务器响应                 │
│  原版客户端：❌ 不可用（无快捷键代码）           │
│                                             │
└─────────────────────────────────────────────┘
```

**工作流程对比：**

**场景 A：右键手持潜影盒（原版客户端可用）**
```
原版客户端                     服务器
     │                             │
     │ 1. 右键（手持潜影盒）           │
     ├──────────────────────────────►│
     │    (原版右键协议)               │ 2. Quick Shulker 拦截
     │                                │ 3. 检测到潜影盒
     │                                │ 4. 打开容器菜单
     │ 5. 接收菜单                    │
     │◄───────────────────────────────┤
     │    (原版菜单协议)               │
```
✅ **原版客户端可用**

**场景 B：快捷键快速打开（原版客户端不可用）**
```
模组客户端                     服务器
     │                             │
     │ 1. 按快捷键                   │
     │ 2. 扫描背包找潜影盒             │
     │ 3. 发送自定义网络包             │
     ├──────────────────────────────►│
     │    (Quick Shulker协议)        │ 4. 接收自定义包
     │                                │ 5. 打开对应容器
     │ 6. 接收菜单                    │
     │◄───────────────────────────────┤

原版客户端
     │
     ❌ 没有快捷键代码
     ❌ 无法发送自定义包
     ❌ 功能不可用
```
❌ **原版客户端不可用**

**Polymer 适配后：**
- ✅ 基础功能：**60-80% 可用**
- ⚠️ 高级功能：**需要模组客户端**
- ⚠️ 需要明确告知用户功能限制

**适配难度：** ⭐ 极简（但需要准确评估）

---

### 类型 3：纯客户端模组

**定义：** 只在客户端运行，不修改服务器逻辑。

**特征：**
- ❓ 完全客户端侧
- ❓ 不涉及服务器同步
- ❓ 不注册自定义内容
- ❓ Polymer 适配无意义

**典型例子：**
- JEI（物品查询）
- Optifine（渲染优化）
- 小地图模组
- 光影模组

**为什么不需要适配？**
```
服务器（原版）                客户端（有模组）
     │                             │
     │                             │ JEI 显示合成表
     │                             │ （本地数据，不通信）
     │                             │
     │                             │ Optifine 优化渲染
     │                             │ （本地渲染，不通信）
     │                             │
     ╰─────────────────────────────╯
            没有注册表同步
            没有自定义内容
            Polymer 无关
```

**判断方法：**
- 如果模组可以在单人世界使用
- 且不影响存档文件内容
- 那就是纯客户端模组

**Polymer 适配：** ❌ **不需要适配**

---

### 类型 4：客户端驱动的服务器模组

**定义：** 有服务器端逻辑，但必须由客户端主动触发才能使用。

**特征：**
- ❌ 核心功能需要客户端代码
- ❌ 使用自定义网络协议
- ❌ 原版客户端无法触发功能
- ❌ Polymer 只能解决连接问题

**典型例子：**
- 仅快捷键驱动的模组
- 自定义 UI 操作的模组
- 客户端配置驱动的功能

**工作流程：**
```
模组客户端                     服务器
     │                             │
     │ 1. 特殊操作（快捷键/UI）       │
     │ 2. 客户端处理                 │
     │ 3. 发送自定义协议包             │
     ├──────────────────────────────►│
     │                                │ 4. 服务器执行
     │                                │
     
原版客户端
     │
     ❌ 没有特殊操作代码
     ❌ 无法发送协议包
     ❌ 完全无法使用
```

**Polymer 适配后：**
- ✅ 可以连接服务器
- ❌ 无法使用任何功能
- ⚠️ 适配价值有限

**适配难度：** ⭐ 极简（但几乎无收益）

---

## 快速判断方法

### 判断流程图

```
开始判断模组类型
    │
    ▼
[模组是否注册自定义内容？]
    │
    ├─ 否 ──► 纯客户端模组 ❓
    │         (不需要适配)
    │
    └─ 是 ──►[功能如何触发？]
              │
              ├─ 原版交互 ──►[数据存储在哪？]
              │  (右键/放置)   │
              │                ├─ 服务器 ──► 纯服务器端模组 ✅
              │                │             (100%可用)
              │                └─ 客户端 ──► 需要深入分析
              │
              ├─ 快捷键+原版 ──► 混合驱动模组 ⚠️
              │                  (60-80%可用)
              │
              └─ 仅快捷键/UI ──► 客户端驱动模组 ❌
                                 (0-20%可用)
```

---

### 3 步快速判断法

#### 步骤 1：查看主类的 `onInitialize()`

```java
// 文件：ModMain.java

@Override
public void onInitialize() {
    // 看这里注册了什么
    Registry.register(...);  // ← 注册了什么？
    
    // 服务器端事件 ✅
    UseBlockCallback.EVENT.register(...);
    
    // 客户端网络包 ⚠️
    ServerPlayNetworking.registerGlobalReceiver(...);
    
    // 仅客户端 ❓
    ClientTickEvents.END_CLIENT_TICK.register(...);
}
```

**判断标准：**
- 只有 `Registry.register` + 服务器事件 → 服务器端模组
- 有网络包注册 → 混合或客户端驱动
- 只有客户端事件 → 纯客户端模组

---

#### 步骤 2：搜索交互代码

```bash
# 在项目中搜索关键词
grep -r "UseBlockCallback" src/
grep -r "UseItemCallback" src/
grep -r "ServerPlayNetworking" src/
grep -r "KeyBinding" src/
```

**判断标准：**
- `UseBlockCallback` 多 → 原版交互为主
- `KeyBinding` 多 → 客户端驱动为主
- 两者都有 → 混合驱动

---

#### 步骤 3：查看 README 或功能描述

**关键词识别：**

✅ **服务器端模组的关键词：**
- "adds custom blocks"
- "right-click to use"
- "place to activate"
- "server-side logic"

⚠️ **混合驱动模组的关键词：**
- "press key to..."
- "hotkey to open"
- "quick access"
- "also works with right-click"

❌ **客户端驱动模组的关键词：**
- "client-side only"
- "requires client mod"
- "hotkey required"
- "custom UI"

---

## 实战案例分析

### 案例 1：Easy Anvils

**快速判断：**

1. **查看注册内容：**
```java
// 注册了自定义方块
Registry.register(Registries.BLOCK, "easyanvils:anvil", CUSTOM_ANVIL);
// 注册了方块实体
Registry.register(Registries.BLOCK_ENTITY_TYPE, ...);
// 注册了菜单类型
Registry.register(Registries.MENU, ...);
```

2. **查看交互方式：**
```java
// 右键方块打开GUI
@Override
public InteractionResult useWithoutItem(...) {
    player.openMenu(...);  // ← 原版交互
    return InteractionResult.SUCCESS;
}
```

3. **数据存储：**
```java
// 物品存储在方块实体
public class AnvilBlockEntity extends BlockEntity {
    private final ItemStackHandler items = ...;  // ← 服务器端数据
}
```

**结论：** ✅ **纯服务器端模组**
- 原版交互（右键方块）
- 服务器端数据（方块实体）
- 原版客户端：**100% 功能可用**

---

### 案例 2：Quick Shulker

**快速判断：**

1. **查看注册内容：**
```java
// 注册了菜单类型
Registry.register(Registries.MENU, "quickshulker:bundle_item", ...);

// 注册了网络包
ServerPlayNetworking.registerGlobalReceiver(OPEN_SHULKER_PACKET, ...);
```

2. **查看交互方式：**
```java
// 方式 1：原版右键
UseItemCallback.EVENT.register((player, world, hand) -> {
    if (stack.getItem() instanceof ShulkerBoxItem) {
        player.openMenu(...);  // ← 原版交互 ✅
    }
});

// 方式 2：客户端快捷键
// 客户端代码：
KeyBinding.onPress(() -> {
    ClientPlayNetworking.send(OPEN_SHULKER_PACKET, ...);  // ← 自定义协议 ❌
});
```

3. **功能分层：**
- 右键功能：原版协议 ✅
- 快捷键功能：自定义协议 ❌

**结论：** ⚠️ **混合驱动模组**
- 基础功能（右键）：原版可用
- 高级功能（快捷键）：需要模组
- 原版客户端：**~60% 功能可用**

## 适配策略选择

### 策略矩阵

| 模组类型 | 适配难度 | 原版功能 | 适配价值 | 推荐策略 |
|---------|---------|---------|---------|---------|
| **纯服务器端** | ⭐⭐⭐ 中等 | 100% | 极高 ⭐⭐⭐⭐⭐ | 完整适配 |
| **混合驱动** | ⭐ 极简 | 60-80% | 中等 ⭐⭐⭐ | 最小适配+说明 |
| **纯客户端** | N/A | N/A | 无 | 不适配 |
| **客户端驱动** | ⭐ 极简 | 0-20% | 低 ⭐ | 不推荐 |

---

### 策略 A：完整适配（服务器端模组）

**适用于：** Easy Anvils 类型

**步骤：**
1. 实现 PolymerBlock 接口（如有自定义方块）
2. 实现 PolymerItem 接口（如有自定义物品）
3. 注册菜单类型和方块实体
4. 测试所有功能

**目标：**
- ✅ 原版客户端 100% 功能可用
- ✅ 视觉效果原版，但功能完整

**文档要求：**
```markdown
## Vanilla Client Support

✅ **Full functionality for vanilla clients**
- All features work exactly the same
- Visual: Vanilla textures
- Functionality: Complete

No client mod required!
```

---

### 策略 B：最小适配+功能说明（混合驱动模组）

**适用于：** Quick Shulker 类型

**步骤：**
1. 仅注册菜单类型（如需要）
2. 不实现复杂接口
3. **重点：明确说明功能限制**

**目标：**
- ✅ 原版客户端可连接
- ⚠️ 部分功能可用（基础交互）
- ❌ 高级功能需要模组

**文档要求：**
```markdown
## Vanilla Client Support

⚠️ **Partial functionality for vanilla clients**

**✅ Works:**
- Right-click held shulker box to open
- Right-click held ender chest to open

**❌ Requires Client Mod:**
- Hotkey to quick-open from inventory
- Hotkey to open player inventory

**For full features, install the mod on client!**
```

---

### 策略 C：不适配（纯客户端模组）

**适用于：** JEI, Optifine 类型

**原因：**
- 不注册服务器端内容
- 不需要注册表同步
- Polymer 适配无意义

**告知用户：**
```markdown
## Installation

**Client-side mod only**
- Install on client for features
- No server installation needed
- Works with any server
```

---

### 策略 D：评估后决定（客户端驱动模组）

**适用于：** 仅快捷键驱动的模组

**评估因素：**
1. 基础功能是否可用？
2. 适配成本 vs 收益
3. 用户期望

**可能结论：**
- 如果有少量原版可用功能 → 最小适配+说明
- 如果完全不可用 → 不适配，说明需要客户端模组

---

## 常见判断错误

### ❌ 错误 1：把所有模组都当成服务器端模组

**症状：**
```markdown
README: ✅ Full functionality for vanilla clients!
实际：原版客户端只能连接，无法使用任何功能
```

**后果：**
- 用户期望与现实不符
- 大量"不工作"的反馈
- 损害项目声誉

**正确做法：**
- 先判断模组类型
- 测试原版客户端功能
- 如实描述可用性

---

### ❌ 错误 2：混淆"可连接"与"可使用"

**症状：**
```
AI: "Polymer 适配成功！原版客户端现在可以连接了！"
用户: "但是我什么功能都用不了？"
```

**问题：**
- Polymer 解决**连接问题**
- 不等于解决**功能可用性**

**正确表述：**
```
✅ "Polymer 适配成功！原版客户端可以连接服务器。"
⚠️ "功能可用性取决于模组类型："
   - "右键功能：可用"
   - "快捷键功能：需要客户端模组"
```

---

### ❌ 错误 3：忽略功能分层

**症状：**
看到模组有网络包就认为原版客户端完全不可用。

**实际：**
Quick Shulker 虽然有自定义网络包（快捷键），但基础功能（右键）仍然可用。

**正确分析：**
```
第 1 层（基础）：
- 右键手持物品 ← 原版协议 ✅

第 2 层（高级）：
- 快捷键快速打开 ← 自定义协议 ❌

结论：60% 可用（不是 0%）
```

---

## 总结与检查清单

### 判断模组类型的检查清单

**步骤 1：注册内容**
- [ ] 检查注册了哪些自定义内容
- [ ] 方块？物品？菜单？实体？
- [ ] 如果什么都没注册 → 纯客户端模组

**步骤 2：触发方式**
- [ ] 查找 `UseBlockCallback`（右键方块）
- [ ] 查找 `UseItemCallback`（右键物品）
- [ ] 查找 `KeyBinding`（快捷键）
- [ ] 查找网络包注册

**步骤 3：功能分层**
- [ ] 哪些功能使用原版交互？
- [ ] 哪些功能需要客户端代码？
- [ ] 比例是多少？

**步骤 4：分类决策**
- [ ] 纯服务器端 → 完整适配
- [ ] 混合驱动 → 最小适配+说明
- [ ] 纯客户端 → 不适配
- [ ] 客户端驱动 → 评估后决定

---

### 适配前必问的 3 个问题

1. **这是什么类型的模组？**
   - 明确分类
   - 不要猜测

2. **原版客户端能用多少功能？**
   - 实际测试
   - 如实评估

3. **如何向用户说明限制？**
   - 清晰描述
   - 避免误导
