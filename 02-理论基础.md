## Polymer 核心原理

### 三层架构详解

#### 第 1 层：注册表隔离

**服务器启动时：**
```
1. 模组注册自定义内容
   Registry.register(Registries.BLOCK, "modid:custom", CUSTOM_BLOCK);

2. Polymer 识别需要隐藏的内容
   - PolymerBlock 实现的方块
   - PolymerMenuUtils 注册的菜单
   - PolymerBlockUtils 注册的 BlockEntity

3. 构建两份注册表
   完整注册表（服务器使用）:
   - minecraft:stone
   - minecraft:dirt
   - modid:custom_block
   
   原版注册表（客户端同步）:
   - minecraft:stone
   - minecraft:dirt
   // modid:custom_block 被隐藏
```

**客户端连接时：**
```
服务器 → 发送原版注册表 → 客户端
客户端对比自己的注册表 → 完全匹配 → 连接成功
```

---

#### 第 2 层：内容转换

**运行时数据包处理：**
```java
// 服务器端
BlockState serverState = level.getBlockState(pos);
// serverState = modid:custom_block[facing=north]

// 发送给客户端前
if (block instanceof PolymerBlock polymerBlock) {
    BlockState clientState = polymerBlock.getPolymerBlockState(serverState);
    // clientState = minecraft:stone[facing=north]
    sendToClient(clientState);
}

// 客户端接收
// 看到: minecraft:stone[facing=north]
// 实际: modid:custom_block[facing=north]（服务器端）
```

**关键点：**
- 服务器内部使用真实方块
- 发送给客户端时动态转换
- 客户端完全不知道有自定义内容

---

#### 第 3 层：数据过滤

**BlockEntity NBT 同步：**
```
服务器端 NBT:
{
  id: "modid:custom_block_entity",
  items: [...],
  customData: {
    energy: 1000,
    temperature: 350
  }
}

Polymer 过滤（PolymerBlockUtils.registerBlockEntity）:
↓

客户端接收 NBT:
{
  id: "minecraft:chest",  // 伪装为原版
  items: [...]            // 保留兼容数据
  // customData 被移除
}
```

## Mixin 软实现原理

### 为什么需要软实现？

**传统接口实现：**
```java
// ❌ 这样做不会修改目标类
@Mixin(CustomBlock.class)
public abstract class CustomBlockMixin implements PolymerBlock {
    @Override
    public BlockState getPolymerBlockState(...) {
        // 这个方法只存在于 Mixin 类
        // 目标类 CustomBlock 并没有实现 PolymerBlock
    }
}
```

**问题：**
```java
// 运行时
CustomBlock block = ...;
if (block instanceof PolymerBlock) {
    // 永远为 false！
    // CustomBlock 类本身没有实现 PolymerBlock
}
```

---

### 软实现机制

**使用 @Implements：**
```java
@Mixin(CustomBlock.class)
@Implements(@Interface(iface = PolymerBlock.class, prefix = "polymer$"))
public abstract class CustomBlockMixin {
    
    public BlockState polymer$getPolymerBlockState(...) {
        return Blocks.STONE.defaultBlockState();
    }
}
```

**字节码转换：**
```
编译前（CustomBlock.class）:
public class CustomBlock extends Block {
    // 原有方法
}

Mixin 处理后:
public class CustomBlock extends Block implements PolymerBlock {
    // 原有方法
    
    // Mixin 注入的接口实现
    public BlockState getPolymerBlockState(...) {
        return this.polymer$getPolymerBlockState(...);
    }
    
    // Mixin 类中的实际实现
    private BlockState polymer$getPolymerBlockState(...) {
        return Blocks.STONE.defaultBlockState();
    }
}
```

**效果：**
```java
// 运行时
CustomBlock block = ...;
if (block instanceof PolymerBlock) {
    // ✅ 现在为 true！
    BlockState polymerState = block.getPolymerBlockState(...);
}
```

---

### 前缀命名规则

**为什么需要前缀？**
```java
// 如果不使用前缀
@Implements(@Interface(iface = PolymerBlock.class, prefix = ""))
public abstract class Mixin {
    public BlockState getPolymerBlockState(...) {
        // 问题：如果目标类已经有同名方法怎么办？
        // 会导致方法冲突！
    }
}

// 使用前缀避免冲突
@Implements(@Interface(iface = PolymerBlock.class, prefix = "polymer$"))
public abstract class Mixin {
    public BlockState polymer$getPolymerBlockState(...) {
        // 方法名唯一，不会冲突
    }
}
```

**Mixin 的转换：**
```
Mixin 类中: polymer$getPolymerBlockState
     ↓ 字节码转换
目标类接口方法: getPolymerBlockState
     ↓ 实际调用
目标类私有方法: polymer$getPolymerBlockState
```

## 注册表同步机制

### Minecraft 注册表系统

**注册表类型：**
```java
// Minecraft 的核心注册表
BuiltInRegistries.BLOCK         // 方块
BuiltInRegistries.ITEM          // 物品
BuiltInRegistries.BLOCK_ENTITY_TYPE  // 方块实体
BuiltInRegistries.MENU          // 菜单类型
BuiltInRegistries.ENTITY_TYPE   // 实体类型
```

**注册过程：**
```java
// 模组注册自定义内容
public static final Block CUSTOM_BLOCK = Registry.register(
    BuiltInRegistries.BLOCK,
    Identifier.fromNamespaceAndPath("modid", "custom_block"),
    new CustomBlock(...)
);

// 结果：CUSTOM_BLOCK 被添加到全局方块注册表
// ID: modid:custom_block
// 数字 ID: 动态分配（如 4567）
```

---

### 客户端连接流程

**完整流程：**
```
1. 客户端发起连接
   ↓
2. 服务器发送 RegistryDataPacket
   包含所有注册表内容:
   {
     blocks: ["minecraft:stone", "minecraft:dirt", "modid:custom_block"],
     items: [...],
     ...
   }
   ↓
3. 客户端接收并对比
   客户端注册表: ["minecraft:stone", "minecraft:dirt"]
   服务器注册表: ["minecraft:stone", "minecraft:dirt", "modid:custom_block"]
   ↓
4. 发现不匹配项: "modid:custom_block"
   ↓
5. 抛出 RemapException
   ↓
6. 断开连接
```

---

### Polymer 的拦截

**Polymer 如何拦截：**
```java
// Polymer 内部伪代码
public class PolymerRegistrySync {
    
    @Inject(method = "sendRegistryData", at = @At("HEAD"))
    private void filterRegistry(ServerPlayer player, CallbackInfo ci) {
        if (isVanillaClient(player)) {
            // 过滤注册表
            filteredRegistry = originalRegistry.filter(entry -> {
                // 移除所有 Polymer 标记的内容
                if (entry instanceof PolymerBlock) return false;
                if (isPolymerMenu(entry)) return false;
                if (isPolymerBlockEntity(entry)) return false;
                return true;
            });
            
            // 发送过滤后的注册表
            sendToClient(player, filteredRegistry);
        } else {
            // 模组客户端：发送完整注册表
            sendToClient(player, originalRegistry);
        }
    }
}
```

**结果：**
```
原版客户端接收:
["minecraft:stone", "minecraft:dirt"]
// modid:custom_block 被过滤

模组客户端接收:
["minecraft:stone", "minecraft:dirt", "modid:custom_block"]
// 完整注册表
```

## 网络协议分析

### 原版协议 vs 自定义协议

**原版协议（所有客户端支持）：**
```java
// 右键方块
UseBlockCallback.EVENT.register((player, world, hand, hitResult) -> {
    // 服务器处理
    player.openMenu(...);
    // 发送原版 OpenScreen 数据包
});

// 客户端接收
// 原版客户端认识 OpenScreen 数据包
// 正常显示 GUI
```

**自定义协议（仅模组客户端支持）：**
```java
// 注册自定义数据包
PayloadTypeRegistry.serverboundPlay().register(
    CustomPacket.ID, 
    CustomPacket.CODEC
);

// 模组客户端发送
ClientPlayNetworking.send(new CustomPacket(...));

// 服务器接收
ServerPlayNetworking.registerGlobalReceiver(CustomPacket.ID, handler);

// 原版客户端
// 不认识 CustomPacket
// 无法发送 → 功能不可用
```

---

### 为什么网络包不影响连接？

**关键区别：**
```
注册表同步（连接时）:
- 发生在连接建立阶段
- 不匹配 → 立即断开
- Polymer 必须处理

网络包通信（运行时）:
- 发生在游戏过程中
- 客户端不发送 → 服务器不处理
- 不影响连接
```

**示例：**
```java
// Quick Shulker 场景

// 原版客户端
玩家按任意键 → 没有 Quick Shulker 代码 → 不发送 OpenShulkerPacket
服务器 → 没收到数据包 → 不执行任何操作
结果: 功能不可用，但不断开连接 ✅

// 模组客户端
玩家按快捷键 → Quick Shulker 拦截 → 发送 OpenShulkerPacket
服务器 → 接收数据包 → 打开容器
结果: 功能正常 ✅
```

## 性能影响分析

### 运行时开销

**注册表隔离（一次性）：**
```
服务器启动: +100ms
客户端连接: +50ms（每个客户端）
```

**方块伪装（每次同步）：**
```java
// 每次发送方块更新
BlockState polymerState = polymerBlock.getPolymerBlockState(state, context);
// 时间: ~1-5 微秒

// 每秒可能的方块更新: 10-100 个
// 总开销: 10-500 微秒 = 0.01-0.5 毫秒
// 相比一帧（16.67ms）：几乎可以忽略
```

**内存开销：**
```
每个 PolymerBlock 实例: +16 bytes（缓存字段）
每个注册表项: +8 bytes（标记）
100 个 Polymer 方块: ~2.4 KB
```

**结论：** 性能影响极小，实际游戏中无法察觉

---

### 网络流量

**方块同步数据包：**
```
原版方块: 4 bytes（方块 ID + 状态）
Polymer 方块: 4 bytes（伪装后的原版 ID）
差异: 0 bytes
```

**BlockEntity 同步：**
```
原版: NBT 大小
Polymer: 过滤后的 NBT（通常更小）
差异: -10% 到 -50%（反而更少！）
```

## 兼容性分析

### 与其他模组的兼容性

**✅ 完全兼容：**
- Fabric API 模组
- 纯服务器端模组
- 不修改注册表的模组

**⚠️ 需要注意：**
- 其他修改注册表的库
- 自定义网络协议的模组
- 需要逐个测试

**❌ 不兼容：**
- 要求所有玩家安装的模组
- 修改核心游戏机制的模组

---

### 与原版的兼容性

**原版客户端保证：**
- ✅ 可以连接
- ✅ 基本游戏功能正常
- ✅ 不会因模组内容崩溃
- ⚠️ 看到的是原版材质

**原版服务器：**
- ❌ Polymer 是 Fabric 库
- ❌ 无法在原版服务器使用
- ✅ 可以在 Fabric 服务器使用

---

## 限制与权衡

### 技术限制

**无法解决的问题：**
1. **自定义材质**
   - 原版客户端没有资源包
   - 只能显示原版材质
   
2. **客户端驱动的功能**
   - 快捷键需要客户端代码
   - Polymer 无法注入代码到原版客户端
   
3. **自定义实体**
   - 渲染需要客户端支持
   - 原版客户端会看到错误的实体

---

### 设计权衡

**选择 Polymer 的好处：**
- ✅ 降低进入门槛
- ✅ 支持混合玩家群体
- ✅ 服务器更新灵活

**付出的代价：**
- ⚠️ 原版客户端视觉效果受限
- ⚠️ 部分功能可能不可用
- ⚠️ 需要额外的开发工作
